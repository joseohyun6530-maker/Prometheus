## 1. 개요

### 1.1 목적

매장 직원이 **주문 접수·제조 진행**을 확인하고, **재고**를 조정하며, 고객이 제출한 주문을 **처리**할 수 있는 **내부용 관리 화면**을 제공한다.

### 1.2 사용자

- 카페 **매장 직원** (주문하기와 동일 디바이스 또는 별도 단말)

### 1.3 앱 내 위치

- 상단 공통 헤더: **COZY**, **주문하기** | **관리자** (관리자 탭 활성)
- **주문하기** 화면에서 제출된 주문·재고와 **API·DB로 연동**
- 탭 전환 시 **주문하기의 장바구니는 유지** (관리자 진입은 `App`에서 `loadAdmin` 호출)

### 1.4 성공 지표 (참고)

- 신규 주문이 주문 현황에 반영됨 (탭 재진입 또는 주문 후 갱신)
- 대시보드 집계가 DB 주문 상태와 일치
- 재고 ±가 즉시 반영되고, 주문하기 화면 품절/담기 규칙과 연동

## 2. 화면 구조 (Information Architecture)

와이어프레임(`wf-2`) 기준 레이아웃:

```
┌─────────────────────────────────────────────────────────┐
│  COZY                          [주문하기] [관리자*]      │  ← 공통 Header
├─────────────────────────────────────────────────────────┤
│  관리자 대시보드                                         │
│  총 주문 n / 주문 접수 n / 제조 중 n / 제조 완료 n        │
├─────────────────────────────────────────────────────────┤
│  재고 현황                                               │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐                 │
│  │ 메뉴명    │ │ 메뉴명    │ │ 메뉴명    │                 │
│  │ 10개     │ │ 10개     │ │ 10개     │                 │
│  │ [-] [+]  │ │ [-] [+]  │ │ [-] [+]  │                 │
│  └──────────┘ └──────────┘ └──────────┘                 │
├─────────────────────────────────────────────────────────┤
│  주문 현황                                               │
│  ┌─────────────────────────────────────────────────┐   │
│  │ 7월 31일 13:00    [주문 접수]                    │   │
│  │ 아메리카노(ICE) x 1              4,000원          │   │
│  │ 주문 금액 4,000원              [제조 시작]       │   │  ← 구현 기준
│  └─────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

### 2.1 영역별 역할

| 영역 | 컴포넌트 | 설명 |
|------|-----------|------|
| 헤더 | `Header` | 관리자 탭 활성 |
| 대시보드 | `AdminDashboard` | 주문 상태별 건수 요약 |
| 재고 | `InventorySection` | 3종 메뉴 재고·± 조정 |
| 주문 | `OrderListSection` | 미완료 주문 목록·상태 전환 |

## 3. 와이어프레임 대비 UI 요구사항

### 3.1 헤더

| 요소 | 요구사항 |
|------|----------|
| 관리자 | 활성 탭 스타일 |
| 주문하기 | 클릭 시 주문 화면 표시 (장바구니 유지) |

### 3.2 관리자 대시보드 (`AdminDashboard`)

와이어프레임 예시: **총 주문 1 / 주문 접수 1 / 제조 중 0 / 제조 완료 0**

| 지표 | 설명 | 데이터 소스 |
|------|------|-------------|
| 총 주문 | DB에 존재하는 **전체** 주문 건수 | `dashboard.total` |
| 주문 접수 | `status === RECEIVED` 건수 | `dashboard.received` |
| 제조 중 | `status === IN_PREPARATION` 건수 | `dashboard.in_preparation` |
| 제조 완료 | `status === COMPLETED` 건수 | `dashboard.completed` |

**표시 형식**

- 한 줄 바: `{라벨} {숫자}` 를 `/` 로 구분
- 구현: `AdminDashboard` — 4개 지표 순서 고정

**비고**

- 목록에 보이는 주문은 **미완료만**이지만, 대시보드 **완료 건수는 전체 DB 기준** (API 집계)

### 3.3 재고 현황 (`InventorySection`)

와이어프레임: **아메리카노(ICE), 아메리카노(HOT), 카페라떼** 3종, 각 **10개**, **+ / −** 버튼

| 요소 | 요구사항 |
|------|----------|
| 대상 메뉴 | `americano-ice`, `americano-hot`, `cafe-latte` 만 |
| 메뉴명 | API `name` |
| 수량 | `{n}개` |
| 상태 뱃지 | `정상` / `주의` / `품절` (구현 확장) |
| − | 재고 1 감소, **0 이하 불가** (0일 때 disabled) |
| + | 재고 1 증가 |

**재고 상태 규칙** (주문하기와 동일)

| 수량 | 라벨 |
|------|------|
| 0 | 품절 |
| 1~4 | 주의 |
| 5+ | 정상 |

**연동**

- 재고 변경 후 `loadMenus` + `loadAdmin`으로 **주문하기** 재고·품절 상태 동기화

### 3.4 주문 현황 (`OrderListSection`)

와이어프레임 예시:

- 시각: `7월 31일 13:00`
- 라인: `아메리카노(ICE) x 1` — `4,000원` (라인 금액)
- 상태/액션: 와이어프레임에 **주문 접수** 버튼 형태 — **구현**은 상태 뱃지 + 액션 버튼 분리

| 요소 | 요구사항 |
|------|----------|
| 목록 범위 | `COMPLETED` **제외** (진행 중 주문만) |
| 정렬 | 주문 일시 **최신순** |
| 시각 | `M월 D일 H:mm` (로컬) |
| 상태 뱃지 | `주문 접수` / `제조 중` / `제조 완료` |
| 라인 | 메뉴명 + 옵션 + `x 수량`, 라인 금액 |
| 주문 금액 | `total_amount` |
| 빈 목록 | "접수된 주문이 없습니다." |

**상태 전환 버튼 (구현)**

| 현재 상태 | 버튼 문구 | 다음 상태 |
|-----------|-----------|-----------|
| `RECEIVED` | **제조 시작** | `IN_PREPARATION` |
| `IN_PREPARATION` | **제조 완료** | `COMPLETED` |
| `COMPLETED` | (목록에 미표시) | — |

와이어프레임의 **주문 접수**는 **상태 표시명**에 해당하며, 다음 단계 액션은 **제조 시작**으로 정의한다.

## 5. 비즈니스 규칙

### 5.1 주문 상태 머신

```
RECEIVED (주문 접수)
    → [제조 시작]
IN_PREPARATION (제조 중)
    → [제조 완료]
COMPLETED (제조 완료)  ← 주문 현황 목록에서 숨김
```

- 역방향 전환·건너뛰기 **불가**
- 고객 주문 생성 시 항상 `RECEIVED`

### 5.2 재고

- 관리자 ±는 **절대값 강제 설정이 아닌 delta** 방식
- `stock_quantity = max(0, current + delta)`
- 주문 생성 시 서버가 자동 차감 (관리자 화면과 별도 트랜잭션)

### 5.3 대시보드 집계

- `total`: `orders` 테이블 전체 row 수
- `received` / `in_preparation` / `completed`: `status`별 `COUNT`
- 활성 목록 필터와 무관하게 **전체 이력** 기준

## 6. API 연동

### 6.1 주문·대시보드

```
GET /api/v1/admin/orders?status=active
```

| 응답 필드 | UI 사용 |
|-----------|---------|
| `orders[]` | 주문 카드 목록 (`getActiveOrders`로 COMPLETED 제외) |
| `orders[].id` | 상태 PATCH path |
| `orders[].ordered_at` / `orderedAt` | 시각 표시 |
| `orders[].status` | 뱃지·버튼 노출 |
| `orders[].lines` | 라인 목록 |
| `orders[].total_amount` / `totalAmount` | 주문 금액 |
| `dashboard.*` | 대시보드 4지표 |

### 6.2 주문 상태 변경

```
PATCH /api/v1/admin/orders/:orderId/status
Content-Type: application/json

{ "status": "IN_PREPARATION" }
```

| HTTP | code | UI |
|------|------|-----|
| 200 | — | 목록 갱신 |
| 400 | `INVALID_REQUEST` | 전환 불가 알림 |
| 404 | `ORDER_NOT_FOUND` | 알림 |

### 6.3 재고

```
GET /api/v1/admin/inventory
```

```json
{
  "items": [
    {
      "menu_id": "americano-ice",
      "name": "아메리카노(ICE)",
      "stock_quantity": 10,
      "stock_status": "정상"
    }
  ]
}
```

```
PATCH /api/v1/admin/inventory/:menuId
{ "delta": 1 }
```

## 9. 와이어프레임 vs 현재 구현 차이

| 항목 | 와이어프레임 (`wf-2`) | 현재 구현 |
|------|----------------------|-----------|
| 대시보드 | 4지표 한 줄 | 동일 |
| 재고 메뉴 | 3종 | 동일 (+ 상태 뱃지) |
| 주문 액션 | "주문 접수" 버튼 형태 | **제조 시작** / **제조 완료** |
| 주문 목록 | 단일 라인 예시 | 복수 라인·옵션·라인별 금액 |
| 완료 주문 | — | 목록에서 숨김, 대시보드 완료 수에는 포함 |
| 실시간 갱신 | — | 탭 진입·액션 후 수동 refetch (폴링 없음) |

향후 와이어프레임과 버튼 문구를 통일하려면 `주문 접수` → `제조 시작` 라벨 매핑 표를 UI 카피에 반영한다.

## 10. 인수 조건 (Acceptance Criteria)

- [ ] 관리자 탭에서 대시보드 4지표가 API와 일치한다.
- [ ] 재고 3종이 표시되고 +/−로 수량이 변경된다.
- [ ] 재고 0에서 − 버튼이 비활성이다.
- [ ] 주문하기에서 주문 후 관리자 주문 현황에 신규 주문이 보인다.
- [ ] **제조 시작** 클릭 시 상태가 제조 중으로 바뀌고 버튼이 **제조 완료**로 바뀐다.
- [ ] **제조 완료** 후 해당 주문이 주문 현황에서 사라지고 완료 건수가 증가한다.
- [ ] 재고 감소 시 주문하기에서 해당 메뉴 품절/담기 제한이 반영된다.
- [ ] 주문하기 탭 전환 후 돌아와도 장바구니가 유지된다.
